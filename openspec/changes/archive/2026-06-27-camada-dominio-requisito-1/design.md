## Context

The `WexTransaction.Domain` project exists as an empty class library. This design introduces the first and only domain aggregate for this application: `PurchaseTransaction`. The domain must enforce all business invariants at construction time, with no dependency on any framework or infrastructure layer.

## Goals / Non-Goals

**Goals:**
- Define the `PurchaseTransaction` aggregate root with a factory method that enforces all field constraints
- Encapsulate the description constraint (max 50 chars) in a `TransactionDescription` value object
- Encapsulate the amount constraint (positive, rounded to nearest cent) in a `Money` value object
- Define typed domain exceptions for each violated invariant, including `InvalidTransactionDateException`
- Enforce UTC-only `DateTimeOffset` for transaction date to guarantee safe Npgsql persistence
- Expose `Money.Value` (decimal) for EF Core conversion with `numeric(18,2)` precision in Infrastructure
- Implement `GlobalExceptionHandler` in `WexTransaction.Api` using `IExceptionHandler` + `AddProblemDetails`
- Produce a fully testable pure-C# domain model

**Non-Goals:**
- EF Core `DbContext`, fluent configuration, or migrations (belong to Infrastructure change)
- Repository interfaces or persistence abstractions (belong to Application layer)
- DTOs or mapping (belong to Application layer)

## Decisions

### 1. Factory method over public constructor

`PurchaseTransaction.Create(description, date, amount)` is the sole creation path, returning a fully valid entity or throwing a domain exception. The parameterless constructor is kept private/protected (required by EF Core for materialization from database, but not for direct instantiation). This enforces the invariant that no `PurchaseTransaction` can exist in an invalid state.

_Alternative considered_: Placing validation in a `Validate()` method called separately â€” rejected because it allows construction of invalid entities.

### 2. Value objects for constrained fields

`TransactionDescription` (string, 1â€“50 chars) and `Money` (decimal, > 0, 2 decimal places) are value objects with implicit conversions to/from their primitive types for ergonomics.

_Alternative considered_: Validating primitives directly in the factory method â€” rejected because it scatters the invariant definition across the codebase as the model grows.

### 3. Rounding strategy for `Money`

`Math.Round(value, 2, MidpointRounding.AwayFromZero)` â€” "nearest cent" per the requirement means 0.005 â†’ 0.01 (away from zero), not banker's rounding (which would give 0.00). This matches standard financial expectations.

### 4. `DateTimeOffset` for transaction date â€” UTC enforced by the domain

`DateTimeOffset` preserves UTC offset information. Npgsql documentation explicitly states:

> *"Npgsql supports reading/writing DateTimeOffset to 'timestamp with time zone' only when the Offset is 0"*

Therefore the factory method validates two conditions: (1) the value is not `default`, and (2) `Offset == TimeSpan.Zero`. Any non-UTC `DateTimeOffset` throws `InvalidTransactionDateException`. This prevents a runtime `InvalidCastException` at the Npgsql persistence layer that unit tests would not detect.

_Alternative considered_: silently normalizing to UTC via `.ToUniversalTime()` â€” rejected because it masks a client error and loses the original offset without feedback.

_Alternative considered_: `DateTime` â€” rejected because it silently loses timezone information when stored in Postgres without a UTC constraint.

### 5. `Guid` as unique identifier

Generated by `Guid.NewGuid()` inside the factory method â€” the domain owns identity creation, not the database. This enables testing without a database and prevents ID collisions across environments.

### 7. `Money` value object and EF Core precision mapping

`Money` exposes a `public decimal Value { get; }` property. The Infrastructure layer will configure EF Core with:

```csharp
entity.Property(e => e.Amount)
    .HasConversion(m => m.Value, v => new Money(v))
    .HasPrecision(18, 2);
```

This maps `Amount` to PostgreSQL `numeric(18, 2)`, guaranteeing exactly 2 decimal places at the database level. The domain already rounds to 2 decimal places before storing, so the database constraint is never violated. No `decimal` precision information is placed inside `WexTransaction.Domain` â€” it belongs exclusively to the Infrastructure fluent configuration.

### 8. Global exception handler in `WexTransaction.Api`

`GlobalExceptionHandler : IExceptionHandler` maps domain exceptions to RFC 7807 `ProblemDetails` using the pattern:

```csharp
// Program.cs
builder.Services.AddExceptionHandler<GlobalExceptionHandler>();
builder.Services.AddProblemDetails();
app.UseExceptionHandler();
```

Mapping strategy:
- `InvalidAmountException`, `InvalidDescriptionException`, `InvalidTransactionDateException` â†’ HTTP 400 with `type`, `title`, and `detail` fields
- Any other `DomainException` subtype â†’ HTTP 422 (Unprocessable Entity)
- Unhandled exceptions â†’ `TryHandleAsync` returns `false`, ASP.NET Core falls back to default ProblemDetails

The handler is registered in `WexTransaction.Api` and has no dependency on domain implementation details â€” it inspects exception type only.

## Risks / Trade-offs

- `Guid.NewGuid()` inside the entity makes the factory non-deterministic â€” unit tests asserting the Id must check it is not `Guid.Empty`, not a specific value â†’ acceptable for domain-level tests
- Value object equality semantics (`record struct` or manual `Equals`) must be consistent to avoid subtle bugs when comparing entities in tests â†’ Mitigation: implement value objects as `readonly record struct`
- `MidpointRounding.AwayFromZero` differs from .NET default banker's rounding â€” any future code that rounds a `Money` value without using the value object will produce different results â†’ Mitigation: all rounding is encapsulated in `Money`; never round outside the value object
- `Money.Value` exposes the internal decimal â€” EF Core queries that filter by amount must use `.Value` directly; the struct is not translatable to SQL as-is â†’ acceptable for this stage; if LINQ-to-SQL queries on amount become necessary, a custom `IRelationalTypeMappingSourcePlugin` can be added later
- `GlobalExceptionHandler` in `WexTransaction.Api` must return `false` from `TryHandleAsync` for unknown exceptions so ASP.NET Core's built-in fallback (500 + ProblemDetails) takes over â†’ Mitigation: always check exception type before handling
