## Purpose

Defines the invariants and construction rules for the `PurchaseTransaction` domain entity. The factory method `PurchaseTransaction.Create` is the single entry point for producing a valid entity; all field constraints are validated atomically before any entity is returned.

## Requirements

### Requirement: Purchase transaction has a valid description
The `PurchaseTransaction` entity SHALL enforce that the description is a non-empty string not exceeding 50 characters. Any attempt to create a transaction with a description that violates this constraint SHALL raise an `InvalidDescriptionException`.

#### Scenario: Valid description is accepted
- **WHEN** `PurchaseTransaction.Create` is called with a description of 1 to 50 characters
- **THEN** the entity SHALL be created with that description stored in `TransactionDescription`

#### Scenario: Empty description is rejected
- **WHEN** `PurchaseTransaction.Create` is called with a null or empty description
- **THEN** an `InvalidDescriptionException` SHALL be thrown

#### Scenario: Description exceeding 50 characters is rejected
- **WHEN** `PurchaseTransaction.Create` is called with a description of 51 or more characters
- **THEN** an `InvalidDescriptionException` SHALL be thrown

#### Scenario: Description of exactly 50 characters is accepted
- **WHEN** `PurchaseTransaction.Create` is called with a description of exactly 50 characters
- **THEN** the entity SHALL be created successfully

### Requirement: Purchase transaction has a valid amount
The `PurchaseTransaction` entity SHALL enforce that the purchase amount is a positive decimal value. The amount SHALL be rounded to the nearest cent (2 decimal places, midpoint away from zero) at the time of creation. Any attempt to create a transaction with a non-positive amount SHALL raise an `InvalidAmountException`.

#### Scenario: Positive amount is accepted and stored rounded to nearest cent
- **WHEN** `PurchaseTransaction.Create` is called with a positive decimal amount (e.g., 200.245)
- **THEN** the entity SHALL be created with the amount rounded to 2 decimal places (e.g., 200.25)

#### Scenario: Amount of zero is rejected
- **WHEN** `PurchaseTransaction.Create` is called with an amount of 0
- **THEN** an `InvalidAmountException` SHALL be thrown

#### Scenario: Negative amount is rejected
- **WHEN** `PurchaseTransaction.Create` is called with a negative amount
- **THEN** an `InvalidAmountException` SHALL be thrown

#### Scenario: Midpoint amount rounds away from zero
- **WHEN** `PurchaseTransaction.Create` is called with an amount whose third decimal digit is 5 (e.g., 1.005)
- **THEN** the stored amount SHALL round up (e.g., 1.01), not use banker's rounding

### Requirement: Purchase transaction has a valid transaction date
The `PurchaseTransaction` entity SHALL enforce that the transaction date is a non-default `DateTimeOffset` value with UTC offset (`Offset == TimeSpan.Zero`). Any attempt to create a transaction with a default or non-UTC date SHALL raise an `InvalidTransactionDateException`. This constraint guarantees safe persistence via Npgsql, which only accepts UTC `DateTimeOffset` for `timestamp with time zone` columns.

#### Scenario: Valid UTC date is accepted
- **WHEN** `PurchaseTransaction.Create` is called with a non-default `DateTimeOffset` with zero offset (UTC)
- **THEN** the entity SHALL be created with that date stored in `TransactionDate`

#### Scenario: Default date is rejected
- **WHEN** `PurchaseTransaction.Create` is called with `DateTimeOffset.MinValue` or `default`
- **THEN** an `InvalidTransactionDateException` SHALL be thrown with a message indicating the date is invalid

#### Scenario: Non-UTC DateTimeOffset is rejected
- **WHEN** `PurchaseTransaction.Create` is called with a `DateTimeOffset` whose `Offset` is not `TimeSpan.Zero` (e.g., `-03:00`)
- **THEN** an `InvalidTransactionDateException` SHALL be thrown

### Requirement: Purchase transaction has a unique identifier assigned at creation
The `PurchaseTransaction` entity SHALL be assigned a non-empty `Guid` as its unique identifier when created via the factory method. The identifier SHALL be generated by the domain and SHALL never be `Guid.Empty`.

#### Scenario: New transaction receives a non-empty Guid
- **WHEN** `PurchaseTransaction.Create` is called with valid inputs
- **THEN** the resulting entity SHALL have an `Id` property that is not `Guid.Empty`

#### Scenario: Two transactions created independently have different identifiers
- **WHEN** `PurchaseTransaction.Create` is called twice with valid inputs
- **THEN** the two resulting entities SHALL have different `Id` values

### Requirement: Purchase transaction factory enforces all invariants atomically
The `PurchaseTransaction.Create` factory method SHALL validate all field constraints before returning an entity. If any constraint is violated, it SHALL throw the appropriate domain exception and SHALL NOT produce a partially constructed entity.

#### Scenario: All valid inputs produce a complete entity
- **WHEN** `PurchaseTransaction.Create` is called with a valid description, valid date, and valid amount
- **THEN** the entity SHALL be fully constructed with all fields populated and a unique Id assigned

#### Scenario: Invalid input prevents entity construction
- **WHEN** `PurchaseTransaction.Create` is called with any invalid input
- **THEN** no entity SHALL be returned and a domain exception SHALL be thrown
